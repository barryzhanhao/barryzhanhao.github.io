MapDB中使用的mmap技术解析

Posted on 2025-05-26 Edited on 2025-05-27

一、背景

在某个公司graalvm项目中，开始使用sqlite作为嵌入式数据库暂存程序运行过程中间态的数据，因需求需要static link静态链接的native二进制文件，sqlite社区不支持，因为当前java版本你的sqlite-jdbc，在static link下，需要动态加载libsqlitejdbc.so库，无法完全static link静态链接。所以尝试更换sqlite，MapDB进入视线，成为备选。

二、MapDB

MapDB简介

以下摘自官网定义

MapDB provides Java Maps, Sets, Lists, Queues and other collections backed by off-heap or on-disk storage. It is a hybrid between java collection framework and embedded database engine. It is free and open-source under Apache license.

MapDB特性

MapDB combines embedded database engine and Java collections. It is free under Apache 2 license. MapDB is flexible and can be used in many roles:

  • Drop-in replacement for Maps, Lists, Queues and other collections.
  • Off-heap collections not affected by Garbage Collector
  • Multilevel cache with expiration and disk overflow.
  • RDBMs replacement with transactions, MVCC, incremental backups etc…
  • Local data processing and filtering. MapDB has utilities to process huge quantities of data in reasonable time.

MapDB支持的特性，就三种in-memory、off-heap 、 on-disk ，完整在org.mapdb.DBMaker中，存储的特性，在org.mapdb.DBMaker.StoreType枚举中

其中，比较有意思的特性是文件存储on-disk中的mmap，也是兼顾可用性和性能的特性。

mmap 的工作原理（底层实现）

首先简单介绍下mmap，memory-mapped内存映射文件的概念和原理。

mmap（memory map）是一种将文件或设备映射到内存的机制，允许程序通过访问内存的方式读写文件，而不是通过传统的 read() / write() 系统调用。

首先简单解释下虚拟内存

虚拟内存（Virtual Memory）是一种由操作系统提供的内存管理机制，它让应用程序“以为”自己拥有一个连续且完整的大内存空间，而实际上：

  • 物理内存（RAM）可能较小；
  • 部分数据可能在磁盘上；
  • 不同程序使用的内存是隔离的。

假设你 mmap 一个 10MB 的文件，操作系统做了这些事情：

1. 分配虚拟地址空间

  • 内核为你的进程分配一段虚拟地址（如 0x10000000 ~ 0x1009FFFF）。
  • 但这段地址还没有真正对应物理内存页（lazy loading）。

2. 页面映射：页缺失时加载

  • 当你第一次访问某个地址，比如 0x10000004，发生页缺失（Page Fault）。
  • 内核捕获后，从文件中加载对应的页（通常 4KB）进物理内存。
  • 更新页表：虚拟页映射到刚刚分配的物理页。

3. 后续访问是直接内存读写

  • 再次访问 0x10000004 就不再发生页缺失，直接读写物理内存。

mmap 的优势总结

优势 说明
性能高 减少系统调用，直接内存访问
零拷贝 不需要将数据从内核拷到用户空间
支持大文件 可以部分映射，按需加载
简化共享 多进程共享内存比管道/消息队列更高效
