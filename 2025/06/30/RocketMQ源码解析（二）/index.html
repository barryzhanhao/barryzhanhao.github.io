RocketMQ源码解析（二）

Posted on 2025-06-30 Edited on 2025-07-03

一、核心概念

##1、概念

以下概念和模型，均抄自RocketMQ官网，RocketMQ的核心概念是清晰和简明的。

Apache RocketMQ 中消息的生命周期主要分为消息生产、消息存储、消息消费这三部分。

生产者生产消息并发送至 Apache RocketMQ 服务端，消息被存储在服务端的主题中，消费者通过订阅主题消费消息。

消息生产

生产者（Producer）：

Apache RocketMQ 中用于产生消息的运行实体，一般集成于业务调用链路的上游。生产者是轻量级匿名无身份的。

消息存储

  • 主题（Topic）：

    Apache RocketMQ 消息传输和存储的分组容器，主题内部由多个队列组成，消息的存储和水平扩展实际是通过主题内的队列实现的。

  • 队列（MessageQueue）：

    Apache RocketMQ 消息传输和存储的实际单元容器，类比于其他消息队列中的分区。 Apache RocketMQ 通过流式特性的无限队列结构来存储消息，消息在队列内具备顺序性存储特征。

  • 消息（Message）：

    Apache RocketMQ 的最小传输单元。消息具备不可变性，在初始化发送和完成存储后即不可变。

消息消费

  • 消费者分组（ConsumerGroup）：

    Apache RocketMQ 发布订阅模型中定义的独立的消费身份分组，用于统一管理底层运行的多个消费者（Consumer）。同一个消费组的多个消费者必须保持消费逻辑和配置一致，共同分担该消费组订阅的消息，实现消费能力的水平扩展。

  • 消费者（Consumer）：

    Apache RocketMQ 消费消息的运行实体，一般集成在业务调用链路的下游。消费者必须被指定到某一个消费组中。

  • 订阅关系（Subscription）：

    Apache RocketMQ 发布订阅模型中消息过滤、重试、消费进度的规则配置。订阅关系以消费组粒度进行管理，消费组通过定义订阅关系控制指定消费组下的消费者如何实现消息过滤、消费重试及消费进度恢复等。

    Apache RocketMQ 的订阅关系除过滤表达式之外都是持久化的，即服务端重启或请求断开，订阅关系依然保留。

2、代码结构

以rocketmq-all-5.3.3版本为例，主要模块在root pom.xm里面的modules，如下图

  • client/

客户端相关代码，包括消息生产者、消费者等客户端 API 实现。

  • common/

公共模块，存放多个模块都会用到的通用工具类、常量、配置等。

  • broker/

Broker 节点的核心实现，负责消息的存储、转发和管理，是 RocketMQ 的核心组件之一。

  • tools/

工具模块，提供命令行工具、运维工具等。

  • store/

消息存储相关实现，负责消息的持久化、检索等。

  • namesrv/

NameServer 实现，负责路由信息的管理和分发，是 RocketMQ 的核心组件之一。

  • remoting/

网络通信相关实现，RocketMQ 内部各组件之间的远程通信基础。

  • srvutil/

服务工具类模块，提供服务启动、关闭等通用工具。

  • filter/

消息过滤相关实现，支持消息的属性过滤等功能。

  • test/

测试模块，集成测试、功能测试等相关代码。

  • distribution/

分发包、脚本、配置文件等，主要用于部署和分发 RocketMQ。

  • openmessaging/

OpenMessaging 兼容层，实现 OpenMessaging 标准的接口。

  • auth/

认证模块，处理 RocketMQ 的身份认证相关逻辑。

  • example/

示例代码，演示如何使用 RocketMQ 的各项功能。

  • container/

Broker 容器化相关实现，支持 Broker 以容器方式运行和管理。

  • controller/

控制器模块，主要用于集群管理、主从切换等高级特性。

  • proxy/

代理模块，提供 RocketMQ 的代理服务，支持多语言接入等。

  • tieredstore/

分层存储模块，实现冷热数据分层存储，提高存储效率。

其中，核心模块，有client、broker、namesrv、remoting、store。其中Producer和Cosumer在client模块，Broker在broker模块，NameServer在namesrv模块。

##3、胖客户端模式

RocketMQ 的逻辑主要分布在 客户端（Producer/Consumer）和服务端（Broker） 两侧，它采取的是一种 “胖客户端 + 精简服务端”的设计思想，这与 Kafka 等“瘦客户端 + 强服务端”有所区别。

客户端主要逻辑

RocketMQ 的 客户端非常“胖”，负责了很多通常服务端承担的责任：

Producer 侧：

  • 路由发现（从 NameServer 主动拉取路由表缓存）
  • 负载均衡（根据 topic 路由选择 queue）
  • 消息发送（支持同步、异步、单向）
  • 自动重试机制（内部维护发送失败重试）
  • 分布式事务消息协调（半消息、本地事务回查）

Consumer 侧：

  • 消费队列负载均衡（客户端做）
  • 消费位点管理（支持本地或 broker 存储）
  • 消息重试机制（客户端拉取失败自动延迟重试）
  • 顺序消费（客户端控制拉取和执行顺序）
  • 消息过滤（tag、SQL 表达式由客户端做二级过滤）

类似于“客户端自行管理状态 + 服务端做存储 + 网络通信”。

服务端主要逻辑（Broker）

RocketMQ 的 Broker 侧负责的是高性能和高可靠的消息存储 + 中转：

  • CommitLog + ConsumeQueue + IndexFile 实现消息存储
  • 顺序写盘 + mmap 实现高吞吐
  • 处理 Producer/Consumer 的请求（写入、读取、确认）
  • 支持消息过滤、过期、重试投递
  • 管理 topic/queue 的元数据
  • 主从复制（同步、异步、DLedger）

虽然它核心是“存储 + 路由转发”，但功能上依然复杂，尤其在：

  • 消息落盘优化（刷盘、堆外内存）
  • 高可用（主从同步、HA 服务）
  • 消费确认（ACK、延迟队列、重试队列）

NameServer 逻辑很轻

  • NameServer 是“无状态”的，只提供“Broker 注册/注销”和“客户端路由发现”
  • 客户端周期性拉取路由缓存
  • Broker 启动时主动注册
  • 多 NameServer 支持容灾（不互通、靠广播）
